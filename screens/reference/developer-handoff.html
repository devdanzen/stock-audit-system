DAO & Transaction Pattern JDBC PreparedStatement 
public List<MasterItem> findByCategory(int categoryId)
        throws SQLException {
    String sql = "SELECT * FROM master_item WHERE category_id = ?";
    List<MasterItem> out = new ArrayList<>();
    try (Connection c = DbConnection.get();
         PreparedStatement ps = c.prepareStatement(sql)) {
        ps.setInt(1, categoryId);
        try (ResultSet rs = ps.executeQuery()) {
            while (rs.next()) out.add(MasterItem.fromRow(rs));
        }
    }
    return out;
}

}
Rules & Best Practices don't skip
  1. try-with-resources for every JDBC object (auto-closes Connection / Statement / ResultSet).
  2. Header + Detail INSERTs always wrapped in setAutoCommit(false) + commit / rollback.
  3. BCrypt only for passwords. Never log password field; only call BCrypt.hashpw() on save and BCrypt.checkpw() on login.
  4. Long DB calls run on SwingWorker, never on the Event Dispatch Thread (UI freeze).
  5. JTable uses DefaultTableModel; per-column DefaultTableCellRenderer sets colored text (red Critical, amber Low, etc.).
  6. ActionListeners wired in a single bindHandlers() method called after initComponents() — never inside the generated GUI builder code.
  7. Validate in the panel (qty > 0, fields not empty) before calling DAO. DAO throws on DB-level violations only.
  8. Dates formatted with DateTimeFormatter.ISO_LOCAL_DATE ("yyyy-MM-dd"). Never SimpleDateFormat.
  9. Sidebar nav swaps screens via cardLayout.show(contentPane, "SalesPanel"). Update active style on the clicked label.
  10. Money columns are DECIMAL(18,4) in DB → BigDecimal in Java. Never double.
When in doubt: Match the existing pattern in MasterItemPanel + MasterItemDAO + MasterItem (View / DAO / Model) — they're the reference triple every other CRUD follows.
